ModelForge
A machine learning pipeline tool I built to solve a problem I kept running into — you have a CSV, you want to know which model works best on it, and you don't want to spend 3 hours writing boilerplate sklearn code to find out.
Live: https://modelforge-g5nw.vercel.app
Backend: https://aman12213-modelforge-backend.hf.space

What it does
You drop a CSV file and ModelForge handles everything from there. It figures out what kind of problem you're dealing with (classification or regression), runs your data through 10 different models with proper cross-validation, tunes the best two with GridSearchCV, and gives you production-ready Python code you can actually run.
The whole thing runs in the browser for the heavy lifting parts like parsing and statistics, and talks to a FastAPI backend on HuggingFace for the actual model training. So the accuracy numbers you see are real — not estimates.

How it works under the hood
When you upload a file, the parser runs reservoir sampling if the file is large. The idea is simple: instead of loading everything into memory, it keeps a running sample where every row has an equal probability of being included. This is how you handle 500MB files without the tab crashing.
For statistics, I used Welford's online algorithm instead of the naive approach. The naive way — sum everything up, divide by n, then do another pass for variance — breaks down with large numbers and requires two passes through the data. Welford does it in one pass with three variables and stays numerically stable throughout.
The backend is a FastAPI server running on Docker. When you kick off the pipeline, your CSV gets sent there, and sklearn does the actual training. LightGBM and XGBoost are both available. The cross-validation is stratified for classification tasks and uses KFold for regression. GridSearchCV runs on the top two models from the benchmark.
The EDA insights and the final report are generated by Claude. Everything else — the stats, cleaning, CV scores, feature importance — is computed from your actual data.

Features
Data Analysis
The statistics panel shows mean, median, standard deviation, skewness, kurtosis, Q1/Q3, IQR, outlier percentage, missing value percentage, and a normality flag based on the Jarque-Bera test. For categorical columns it shows cardinality, Shannon entropy, class imbalance ratio, and recommended encoding strategy.
Data Quality Engine
Automatically flags leaky columns (anything that looks like an ID or hash), constant columns, high missing rate, severe class imbalance, and high cardinality. Each issue gets a severity level and a suggested fix. The health score is a 0 to 100 number that rolls all of these signals into one.
Data Cleaning
You can apply fixes directly in the browser — fill missing values with mean, median, mode, or zero, cap outliers at the IQR bounds, drop rows with missing values, or drop a column entirely. After applying, you get a preview and can download the cleaned CSV.
Model Benchmark
Runs 10 models for classification (LightGBM, XGBoost, RandomForest, GradientBoosting, ExtraTrees, LogisticRegression, SVC, KNN, DecisionTree, GaussianNB) or 10 for regression (LightGBM, XGBoost, RandomForest, GradientBoosting, ExtraTrees, Ridge, Lasso, ElasticNet, SVR, DecisionTree). All with 5-fold stratified cross-validation.
Hyperparameter Tuning
GridSearchCV on the top two models. The search grids cover the parameters that actually matter for each model type — learning rate and tree depth for boosting models, regularization strength for linear models.
Live Prediction
After training, you can enter feature values manually and get a prediction from the best model. Categorical columns show a dropdown of the most common values. Numeric columns default to the training mean.
Python Code Export
Generates a complete sklearn pipeline — ColumnTransformer with separate preprocessing for numeric and categorical columns, cross-validation, GridSearchCV, test set evaluation, feature importance extraction, MLflow logging, and a joblib export. It uses the actual column names and types from your dataset.

Running locally
Clone the frontend:
git clone https://github.com/aman-pathak1/modelforge
cd modelforge
npm install
npm run dev
Clone the backend:
git clone https://github.com/aman-pathak1/modelforge-backend
cd modelforge-backend
pip install -r requirements.txt
uvicorn main:app --reload
The frontend expects the backend at the HuggingFace URL by default. If you're running the backend locally, change the BACKEND constant in App.jsx to http://localhost:8000.

Stack
Frontend is React with Vite. Charts are Recharts. No component library — all styles are written inline. The backend is FastAPI with scikit-learn, LightGBM, XGBoost, and pandas. Deployed on Vercel (frontend) and HuggingFace Spaces via Docker (backend).

Known limitations
The HuggingFace free tier puts the container to sleep after inactivity. First request after a sleep period takes 30 to 60 seconds to respond. Subsequent requests are fast. If you're running this seriously you'd want to host the backend somewhere that doesn't sleep.
Training time depends on dataset size. For datasets under 10k rows most models finish in under 30 seconds total. Larger datasets take longer, especially GradientBoosting and SVC.
The live prediction endpoint retrains the model on every request which is fine for a demo but not how you'd do it in production.
